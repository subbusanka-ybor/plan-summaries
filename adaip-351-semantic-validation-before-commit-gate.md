# ADAIP-351: Semantic Validation Before Commit Gate

## Context

Cortex generates deployment manifests (K8s, CI, Dockerfile) from static templates with variable substitution. The existing `CommitValidator` enforces **structural** correctness — directory layout, required fields, IMAGE_NAME suffix rules. But it cannot catch **semantic** mismatches between generated files and the actual source code.

This gap was discovered reviewing cortex-mcp PR #2 (generated by Cortex for the cortex-mcp service). The templates produced correct output for standard apps (e.g., sale-service), but for non-standard apps the defaults went unchallenged:
- Health probe defaulted to `/health/ready` — but the app exposes `/health`
- Port defaulted to `8080` — correct for this case, but unverified
- Config block was empty — but the app uses several env vars (`DATABASE_URL`, `ANTHROPIC_API_KEY`, etc.)

These aren't template bugs — the templates correctly use defaults. The gap is that nobody cross-references the generated output against what the source code actually does.

**Goal**: Add a semantic validation pass that reads the source repo and warns about mismatches. Findings are **warnings only** (never block commits) — they surface as advisory messages so Claude can self-correct before the developer reviews the PR.

## Approach

Add a `SemanticValidator` class in `commit_validator.py` that runs after structural validation passes. It:
1. Parses the generated manifests (already available as `files` in the commit)
2. Reads the source repo via `GitHubClient` to understand what the app actually does
3. Returns `Violation` objects with `severity="warning"` — appended to commit output, never blocking

### Integration point

In `CommitGitHubFiles.execute()` (`github_write.py`), after the structural `CommitValidator` passes and before `client.commit_files()`:

```python
# Semantic validation (advisory — never blocks)
semantic = SemanticValidator()
semantic_result = await semantic.validate(
    files=files,
    owner=owner,
    repo=repo,
    context=context,
)
if semantic_result.violations:
    # Append warnings to commit output (shown to Claude after commit)
    semantic_warnings = semantic_result.format_warnings()
```

## Checks

### Check 1: Health probe path verification
- Parse generated `application.yaml` for `readinessProbe` → `path`
- Fetch the repo tree via `GitHubClient.get_tree_recursive()`
- Scan Python files for route decorators matching the health path (`@app.get("/health/ready")`, `@router.get(...)`, Flask `@app.route(...)`)
- Also check for common health frameworks: `fastapi_health`, `django-health-check`
- **Warn** if the declared health path doesn't appear in any route decorator

### Check 2: Port consistency
- Parse generated `application.yaml` for `deployment.ports[0].port`
- Parse the Dockerfile (if in the commit files) for `EXPOSE` directives and CMD `--port` args
- If both exist, compare — **warn** on mismatch
- If only manifest port exists, check source code for common port patterns (`uvicorn ... --port`, `app.run(port=...)`, `EXPOSE`)

### Check 3: Config completeness (env var coverage)
- Parse generated `application.yaml` `spec.config` section for declared env var names
- Use existing `_parse_os_environ_refs()` from `detect_secrets_usage.py` to find env vars actually referenced in source code
- Cross-reference: **warn** about env vars used in source but not declared in config or secrets
- Filter out standard runtime vars (`PATH`, `HOME`, `PYTHONPATH`, `NODE_ENV`, etc.)

## Implementation Details

### `SemanticValidator` class (in `commit_validator.py`)

```python
class SemanticValidator:
    """Cross-references generated manifests against source repo."""

    async def validate(
        self,
        files: list[dict],
        owner: str,
        repo: str,
        context: dict | None = None,
    ) -> ValidationResult:
        """Run semantic checks. All findings are warnings."""
        ...
```

Key design decisions:
- **Single `get_tree_recursive()` call** shared across all checks (one API call, not one per check)
- **Selective file reads** — only fetch files that match patterns (e.g., `*.py` for route scanning), limit to ~10 files max to avoid API rate limits
- **Reuses existing patterns**: `_resolve_token()` from `github_read.py` for token selection, YAML parsing pattern from `KubernetesRuleSet`
- **Reuses `_parse_os_environ_refs()`** from `detect_secrets_usage.py` for env var extraction (no duplication)
- **Graceful degradation** — if source repo is inaccessible (token issues, private repo), skip semantic checks silently with a log message

### Route detection patterns (Check 1)

For Python apps, scan for these patterns:
```python
_ROUTE_PATTERNS = [
    re.compile(r'@\w+\.(get|post|put|delete|head|options|patch)\s*\(\s*["\']([^"\']+)'),  # FastAPI/Flask decorators
    re.compile(r'\.add_route\s*\(\s*["\']([^"\']+)'),  # Starlette add_route
    re.compile(r'\.add_api_route\s*\(\s*["\']([^"\']+)'),  # FastAPI add_api_route
    re.compile(r'app\.include_router.*prefix\s*=\s*["\']([^"\']+)'),  # Router prefix
]
```

### Warning format

```
SEMANTIC REVIEW (advisory — these did not block the commit):
  1. [health-probe] Health path '/health/ready' not found in source code routes.
     Detected routes: /health, /api/v1/users, /api/v1/items
     Consider updating readinessProbe.path to '/health' in application.yaml.
  2. [config] Source code references env vars not declared in manifest config:
     DATABASE_URL, ANTHROPIC_API_KEY, OPENAI_API_KEY
     Consider adding these to spec.config or spec.secrets in application.yaml.
```

## What This Does NOT Do

- Does NOT block commits (all findings are `severity="warning"`)
- Does NOT modify generated files (validator is read-only)
- Does NOT replace structural validation (runs in addition to it)
- Does NOT require new API keys or tools
- Does NOT affect the render pipeline — validation is post-render, pre-commit
